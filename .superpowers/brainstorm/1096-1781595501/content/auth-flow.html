New Authentication & Onboarding Flow

Three distinct paths — initial onboarding, standard login, admin reset

① Initial Onboarding

1
Admin invites user inviteUserByEmail → Supabase sends OTP email
Unchanged
↓
2
User clicks OTP link Lands on /auth/callback with code
Modified
↓
3
Callback checks password_set users.password_set = false → redirect to /login/set-password
Modified
↓
4
User sets password 8–16 chars, 3 of 4 rules, confirmed twice
New page
↓
5
Password saved updateUser({password}) + password_set = true → /timesheet/daily

② Standard Login

1
User opens /login Email + Password form (replaces OTP input)
Modified
↓
2
signInWithPassword Supabase validates credentials
Modified
↓
3
Redirect to /timesheet/daily Normal authenticated session
No "Forgot password" link
Password reset is admin-only. Login page has no self-service option.

③ Admin Password Reset

1
Admin clicks "Reset Password" Button on user list row AND user edit page
New action
↓
2
Server action runs ① set password_set = false
② revoke user sessions (service client)
③ send recovery email via Admin API
↓
3
User logged out immediately Any active session is invalidated
↓
4
User clicks recovery link /auth/callback → password_set = false → /login/set-password
↓
5
Repeats onboarding flow Same password creation screen as initial setup
One DB change required ALTER TABLE users ADD COLUMN password_set boolean NOT NULL DEFAULT false; — all existing users start as false (but only matter when they next need to authenticate via OTP/recovery link).
